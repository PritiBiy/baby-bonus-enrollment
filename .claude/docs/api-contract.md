# API Contracts

> Reference document for Claude when building controllers, DTOs, and integration tests.
> All NRIC values in responses must be masked. All error responses use `ErrorResponse`.

---

## Summary Table

| Method | Path | Description                                 |
|--------|------|---------------------------------------------|
| `POST` | `/api/v1/enrollments` | Submit an enrollment application            |
| `GET` | `/api/v1/enrollments/{id}` | Retrieve enrollment by ID with disbursement 
| `GET` | `/api/v1/enrollments?childNric=` | List all enrollment rekcords for a child    |
| `PATCH` | `/api/v1/enrollments/{id}/ineligible` | Override enrollment status to INELIGIBLE    |

---

## Common Structures

### ErrorResponse
All error responses use this shape:
```json
{
  "error": "<human-readable message>"
}
```
Never expose: stack traces, class names, SQL errors, or internal field names.

### NRIC Masking
- Format: first 4 characters + `****` + last character
- Example: `T2400001A` → `T240****A`
- Apply to **all** NRIC fields in **all** responses — both `childNric` and `parentNric`
- Apply in audit logs too — never log a raw NRIC

### Authentication
All endpoints require:
```
X-API-Key: <your-api-key>
```
Missing or invalid key → `401 Unauthorised`

## POST /api/v1/enrollments

Submit an enrollment application for a child. Runs eligibility checks against ICA and IROAS stubs. On success, creates an enrollment record and initiates a cash gift disbursement request of $3,000.

> **Note on disbursement status:** In production, the disbursement service generates the disbursement ID and processes it asynchronously. The POST response therefore returns disbursement status as `PENDING` with `processedAt: null`. The caller should poll `GET /api/v1/enrollments/{id}` to observe the final disbursement status.
>
>[**PENDING TO DECIDE**] In this implementation, disbursement is processed synchronously within the same transaction as an explicit scope cut. The response will show `PROCESSED` and a populated `processedAt` immediately.

### Request Body

```json
{
  "childNric": "T2400001A",
  "parentNric": "S8001234A",
  "relationship": "FATHER"
}
```

| Field | Type | Validation | Notes |
|-------|------|------------|-------|
| `childNric` | String | `@NotBlank` | NRIC of the child being enrolled |
| `parentNric` | String | `@NotBlank` | NRIC of the applying parent or guardian |
| `relationship` | Enum | `@NotNull` | One of: `FATHER`, `MOTHER`, `LEGAL_GUARDIAN` |

### Success Response — 201 Created

```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "childNric": "T240****A",
  "parentNric": "S800****A",
  "relationship": "FATHER",
  "status": "ENROLLED",
  "enrolledAt": "2024-08-01T10:00:00Z",
  "disbursement": {
    "id": "b2c3d4e5-f6a7-8901-bcde-f12345678901",
    "type": "CASH_GIFT",
    "amount": 3000.00,
    "status": "PENDING",
    "processedAt": null
  }
}
```

> In this implementation `status` will be `PROCESSED` and `processedAt` will be populated immediately due to the synchronous scope cut.

| Field | Type | Notes |
|-------|------|-------|
| `id` | UUID | Enrollment ID |
| `childNric` | String | Masked |
| `parentNric` | String | Masked |
| `relationship` | Enum | Echoed back from request |
| `status` | Enum | Always `ENROLLED` on 201 |
| `enrolledAt` | Instant | ISO-8601 UTC |
| `disbursement.id` | UUID | Disbursement ID — generated by disbursement service in production; generated locally in this implementation |
| `disbursement.type` | Enum | Always `CASH_GIFT` for now |
| `disbursement.amount` | Decimal | Always `3000.00` for now |
| `disbursement.status` | Enum | `PENDING` in production model; `PROCESSED` in this implementation |
| `disbursement.processedAt` | Instant | `null` on 201 in production model; populated immediately in this implementation |

### Error Responses

| Scenario | Status | Error Message |
|----------|--------|---------------|
| Child not found in ICA | 422 | `Child not found in ICA records` |
| Child is not a Singapore Citizen | 422 | `Child is not a Singapore Citizen` |
| Parent not found in IROAS | 422 | `Parent not found in IROAS records` |
| Child already has an active enrollment | 409 | `Child already has an active enrollment` |
| Any request field blank or null | 400 | `<fieldName> must not be blank` |
| Invalid relationship value | 400 | `Invalid value for relationship` |
| Missing or invalid API key | 401 | `Unauthorised` |

## GET /api/v1/enrollments/{id}

Retrieve a single enrollment record by its UUID, including disbursement details. In the production async model, callers should poll this endpoint to observe the final disbursement status after the initial `PENDING` returned by POST.

### Path Parameter

| Parameter | Type | Notes                                                |
|-----------|------|------------------------------------------------------|
| `id` | UUID | Enrollment ID returned from register enrollment call |

### Success Response — 200 OK

```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "childNric": "T240****A",
  "parentNric": "S800****A",
  "relationship": "FATHER",
  "status": "ENROLLED",
  "enrolledAt": "2024-08-01T10:00:00Z",
  "disbursement": {
    "id": "b2c3d4e5-f6a7-8901-bcde-f12345678901",
    "type": "CASH_GIFT",
    "amount": 3000.00,
    "status": "PROCESSED",
    "processedAt": "2024-08-01T10:00:01Z"
  }
}
```

`disbursement` may be `null` if the enrollment is `INELIGIBLE` and no disbursement was created.

### Error Responses

| Scenario | Status | Error Message |
|----------|--------|---------------|
| Enrollment ID not found | 404 | `Enrollment not found` |
| Missing or invalid API key | 401 | `Unauthorised` |

## GET /api/v1/enrollments?childNric=

List all enrollment records for a given child NRIC. A child may have multiple records across different statuses (e.g. an earlier `INELIGIBLE` attempt followed by a successful `ENROLLED`). Returns an empty array if no records exist — not a 404.

### Query Parameter

| Parameter | Type | Validation | Notes |
|-----------|------|------------|-------|
| `childNric` | String | Required | Raw (unmasked) NRIC — this is the lookup key sent by the caller |

### Success Response — 200 OK

```json
[
  {
    "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "childNric": "T240****A",
    "parentNric": "S800****A",
    "relationship": "FATHER",
    "status": "ENROLLED",
    "enrolledAt": "2024-08-01T10:00:00Z",
    "disbursement": {
      "id": "b2c3d4e5-f6a7-8901-bcde-f12345678901",
      "type": "CASH_GIFT",
      "amount": 3000.00,
      "status": "PROCESSED",
      "processedAt": "2024-08-01T10:00:01Z"
    }
  },
  {
    "id": "c3d4e5f6-a7b8-9012-cdef-123456789012",
    "childNric": "T240****A",
    "parentNric": "S820****B",
    "relationship": "MOTHER",
    "status": "INELIGIBLE",
    "enrolledAt": null,
    "disbursement": null
  }
]
```

### Error Responses

| Scenario | Status | Error Message |
|----------|--------|---------------|
| `childNric` query param missing | 400 | `childNric query parameter is required` |
| Missing or invalid API key | 401 | `Unauthorised` |

## PATCH /api/v1/enrollments/{id}/ineligible

Ops function to manually override an enrollment to `INELIGIBLE` with a reason. Used when source data was incorrect after the fact. Can override any status including `ENROLLED`.

### Path Parameter

| Parameter | Type | Notes |
|-----------|------|-------|
| `id` | UUID | Enrollment ID to override |

### Request Body

```json
{
  "reason": "Child citizenship data incorrect in ICA at time of enrollment"
}
```

| Field | Type | Validation | Notes |
|-------|------|------------|-------|
| `reason` | String | `@NotBlank` | Human-readable reason for the override, stored on the record |

### Success Response — 200 OK

```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "childNric": "T240****A",
  "parentNric": "S800****A",
  "status": "INELIGIBLE",
  "reason": "Child citizenship data incorrect in ICA at time of enrollment"
}
```

Only the fields relevant to the override are returned — no disbursement details. Disbursement reversal is a separate refund flow (deferred, out of scope).
<br/>
Future scope: Reason can be populated in dropdown to select some presets and allow to enter new one as needed. This will help to create dashboard by grouping error reason 

### Error Responses

| Scenario | Status | Error Message |
|----------|--------|---------------|
| Enrollment ID not found | 404 | `Enrollment not found` |
| Enrollment already INELIGIBLE | 422 | `Enrollment is already ineligible` |
| `reason` blank or missing | 400 | `reason must not be blank` |
| Missing or invalid API key | 401 | `Unauthorised` |
